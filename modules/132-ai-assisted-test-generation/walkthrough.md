# AI-Assisted Test Generation & Snapshot Testing - Hands-on Walkthrough

In this walkthrough, you'll add two layers of protection to your PoC: AI-generated unit tests for critical functions, and a snapshot testing setup that uses Git as your assertion engine. You'll break something intentionally, see the snapshot catch it, and learn the accept-or-reject workflow.

## Prerequisites

See [module overview](about.md) for full prerequisites list.

## What We'll Build

By the end of this walkthrough you will have:

- **AI-generated unit tests** — a `tests/` folder with tests for at least one function in your PoC, generated by AI and verified by you
- **A snapshot runner script** — a Python (or Node.js) script that executes your application as a black box and dumps results to `.snapshot` files
- **A committed baseline snapshot** — the `.snapshot` files committed to Git, forming your regression baseline
- **A verified catch** — hands-on proof that the snapshot catches a real change and `git diff` shows exactly what broke

## Part 1: Understand the two testing approaches

Before writing any code, ask AI to explain the two approaches you'll use today:

```
Explain the difference between unit testing and snapshot testing
for a non-programmer manager. Use this example: I have a Python 
function that calculates user access permissions based on their role.
What would a unit test look like? What would a snapshot test look like?
Which is easier to set up? Which catches more types of bugs?
```

Read the response. The key insight:
- **Unit tests** — you write explicit assertions (`assertEqual(result, "admin")`) for specific inputs
- **Snapshot tests** — you run the whole thing, dump the output, commit it; next time it runs and the output differs, you know something changed

Snapshot testing requires zero understanding of the code internals. That makes it perfect for AI-generated code.

## Part 2: Generate unit tests with AI

Pick one function from your PoC — ideally one that takes inputs and returns a predictable output (a calculation, a formatter, a validator).

Show it to AI and ask for tests:

```
Here is a function from my application:

[paste the function code here]

Generate unit tests for this function using [pytest for Python / Jest for Node.js].
Cover:
1. The happy path (normal inputs, expected output)
2. Edge cases (empty input, zero, maximum values)
3. Error cases (what happens with wrong types or invalid values)

Write the tests as a complete file I can run immediately.
Include a comment explaining what each test is checking and why.
```

Save the file in a `tests/` folder:

**Windows:** `c:/workspace/hello-genai/tests/test_function_name.py`  
**macOS/Linux:** `~/workspace/hello-genai/tests/test_function_name.py`

Run the tests:

**Python:**
```
pip install pytest
pytest tests/
```

**Node.js:**
```
npm install --save-dev jest
npx jest
```

**What just happened:** All tests should pass immediately — because AI generated them to match the current behaviour. The value isn't that they pass now — it's that they'll fail if someone changes the function behaviour later.

If any test fails, ask AI: "This test is failing with [error]. Is the test wrong or is the function wrong? Explain."

## Part 3: Write the snapshot runner

**What we're about to do:** Create a script that calls your application like a real user would — sending HTTP requests, running CLI commands, or calling functions directly — and saves the output to a file. This file becomes your regression baseline.

Ask AI to generate the runner for your specific application:

```
I have a [describe your app: web API / CLI tool / Python module].
Here are the main operations it performs: [list 3-5 operations].

Generate a Python script called snapshot_runner.py that:
1. Calls each operation with sample inputs
2. Collects the output (response body, stdout, or return value) for each
3. Saves all outputs to a file called snapshot_current.txt
   - Format: one labelled section per operation
   - Use clear separators so differences are easy to read in git diff
4. Prints "Snapshot saved to snapshot_current.txt" when done

Use the requests library if my app is an HTTP API.
```

Save the script to your project root. Run it:

```
python snapshot_runner.py
```

Open `snapshot_current.txt`. Read through it. This is your application's current output — every single operation, captured.

---

## Part 4: Commit the baseline snapshot

This is the most important conceptual step. The snapshot file IS the test. Committing it is the act of saying "this is correct behaviour."

```
git add snapshot_current.txt
git commit -m "test: add snapshot baseline for regression testing"
```

**What just happened:** Git now knows what your application output looks like. Any future change to the output will show up in `git diff`. The snapshot file is now your assertion.

Rename the file to make it clear it's the reference:

```
# Windows
rename snapshot_current.txt snapshot_baseline.txt

# macOS/Linux
mv snapshot_current.txt snapshot_baseline.txt
```

Update your snapshot_runner.py to save to `snapshot_current.txt` (not baseline — you'll compare against baseline).

---

## Part 5: Break something and catch it

**What we're about to do:** Change your code deliberately — a small, intentional modification — and watch the snapshot catch the regression.

Pick any small change:
- Change a default value (e.g., `max_results = 10` to `max_results = 5`)
- Add a field to an output
- Change a response format

After making the change, run the snapshot runner again:

```
python snapshot_runner.py
```

Now compare to the baseline:

```
git diff snapshot_baseline.txt snapshot_current.txt
```

You should see exactly what changed — lines removed in red, lines added in green.

**This is the full workflow.** You now have two choices:

1. **The change is a regression (unintentional)** → Revert your code: `git checkout -- [filename]`
2. **The change is intentional** → Accept the new snapshot:
   ```
   copy snapshot_current.txt snapshot_baseline.txt   # Windows
   cp snapshot_current.txt snapshot_baseline.txt     # macOS/Linux
   git add snapshot_baseline.txt
   git commit -m "test: update snapshot — changed max_results to 5"
   ```

## Part 6: Add multiple scenarios

One snapshot isn't enough. Add scenarios for different inputs — edge cases, different users, error paths.

Ask AI to extend your runner:

```
Add these additional scenarios to snapshot_runner.py:
1. [describe a different user type or input]
2. [describe an edge case — empty list, zero, missing field]
3. [describe an error case — invalid token, 404 path]

For each scenario, add a clearly labelled section to the output file:
=== SCENARIO: [name] ===
[output]
```

Run the extended runner. Review the new snapshot sections. Commit the baseline again.

## Understanding Check

1. **Why is committing the snapshot file equal to "writing an assertion"?**
   > Git stores the exact state of the file. When you run the snapshot again and compare with `git diff`, any difference is a failed assertion — something the application now does differently. The commit is the "expected value"; the new run is the "actual value".

2. **What is the main limitation of snapshot testing compared to unit testing?**
   > Snapshot tests don't tell you *why* something changed — only *that* it changed. If the output changes for a good reason (new feature), you must update the baseline. They also don't catch bugs where the output format stays the same but the logic is silently wrong.

3. **When should you accept a snapshot update vs. revert the code?**
   > Accept when the change is intentional and the new output is correct (e.g., you added a new field deliberately). Revert when the change is unexpected — you changed something unrelated and the snapshot changed, meaning your change had unintended side effects.

4. **Why are unit tests generated by AI usually correct immediately (all tests pass on the first run)?**
   > AI generates tests to match the current behaviour of the function. It reverse-engineers tests from implementation. This means the tests pass — but they also mean the tests only protect against future changes, not current bugs. To find current bugs, you'd need to write tests based on requirements, not code.

5. **A colleague says: "The snapshot file keeps changing in git, it's annoying noise." How do you respond?**
   > The snapshot changing is a signal, not noise. It means the application output changed. The workflow is: see the diff → decide if it's intentional → either accept (update baseline) or revert. If snapshots change too often, the underlying code is too unstable — that's the real problem to fix.

6. **What should the snapshot runner capture for an HTTP API application?**
   > At minimum: status code, response body, and key headers for each endpoint call. Optionally: response time (for performance regression detection). The output should be stable between runs — avoid timestamps or random IDs in the snapshot output or normalize them first.

7. **What is the path to integrating snapshot tests into CI/CD (module 155)?**
   > In the CI pipeline, the snapshot runner runs against the deployed application and its output is compared to the committed baseline. If they differ, the pipeline fails. This turns "compare manually with git diff" into an automated gate on every deployment.

## Troubleshooting

**Tests fail because of timestamps or random IDs in output**
> These make snapshots non-deterministic — they'll always be different. Ask AI: "Modify the snapshot runner to replace timestamps with [TIMESTAMP] and UUIDs with [UUID] before saving." This normalises the output so only meaningful changes appear.

**pytest says "no tests found"**
> Ensure test file names start with `test_` and test function names start with `test_`. Pytest only discovers tests that follow this convention. Run `pytest --collect-only` to see what it finds.

**`git diff` shows the entire file changed (whitespace only)**
> Git might be normalising line endings. Run `git diff --ignore-space-change snapshot_baseline.txt snapshot_current.txt` to filter out whitespace-only differences.

**AI-generated tests use imports that don't exist**
> AI may reference the function by incorrect module path. Check the actual path to your function and update the import line at the top of the test file. Then ask AI: "Fix the imports — my module is located at [actual path]."

**The snapshot runner hangs on startup**
> It's probably waiting for the application server to start. Make sure your server is running before you execute the runner. Or ask AI to add a startup check: "Add code to verify the server is responding before running scenarios."
