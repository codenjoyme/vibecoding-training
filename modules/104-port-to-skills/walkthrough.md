# Port Existing Code into Skills - Hands-on Walkthrough

In modules 100 and 103 you worked with three tools: `echo`, `get_time`, and `calculate`. In module 100 they lived inside an MCP server. In module 103 they lived inside a REST server. Now you'll port the same logic one more time — but this time the reference is a Java Spring service, and the output is a standalone CLI script packaged as a reusable AI Skill.

The tools are deliberately the same. This lets you focus on *the porting technique*, not on understanding new domain logic. By the end, you'll have the same three operations available as a CLI tool that requires no server, no protocol, and zero LLM involvement to produce results.

## Prerequisites

- Complete [AI Skills & Tools Creation](../090-ai-skills-tools-creation/about.md) — you must understand the Skill formula (instruction + tool)
- Complete [CLI: Command Line Interface](../103-cli-command-line-interface/about.md) — you must understand why CLI bypasses LLM for deterministic results
- Python 3 available in your terminal (`python --version` or `python3 --version`)

| Artifact | Path | Purpose |
|---|---|---|
| Reference Java service | `tools/reference/ToolsService.java` | Source of truth for logic (already provided — do not run) |
| Python CLI tool | `tools/tools_cli.py` | Cross-platform script generated by AI from the Java reference |

The Java service is a **mock reference** — you never compile or run it. Its value is purely as a specification that AI can read and port.

## Part 1: Understand What You're Porting

### Step 1.1 — Open the reference service

Open `./tools/reference/ToolsService.java`.

Read it and notice:
- Three public methods: `echo`, `getTime`, `calculate`
- `echo` returns `"Echo: " + text` — same prefix as in module 100 and 103
- `getTime` returns `"Current time: yyyy-MM-dd HH:mm:ss"` — same format as before
- `calculate` supports `add`, `subtract`, `multiply`, `divide` — same four operations
- Output format strings are specified exactly: `"Result: {a} {operation} {b} = {result}"`
- Edge cases: division by zero and unknown operation throw `IllegalArgumentException`

**Key insight:** The Javadoc and implementation together are a complete specification. You don't need a separate requirements document — the code *is* the requirements. The LLM can read it and port it without any guessing.

### Step 1.2 — Identify the CLI shape

From the Java service, the CLI needs three subcommands:

| Subcommand | Parameters |
|---|---|
| `echo` | `--text` |
| `get-time` | *(no parameters)* |
| `calculate` | `--a`, `--b`, `--operation` |

You now have everything you need. Time to delegate.

## Part 2: Ask AI to Port the Logic

Open your AI assistant with Agent Mode enabled (Claude Sonnet 4.5 recommended).

### Step 2.1 — Give AI the reference and request porting

Use this prompt:

```
I have a Java Spring service at ./modules/104-port-to-skills/tools/reference/ToolsService.java.

Port its logic to a Python CLI script at ./modules/104-port-to-skills/tools/tools_cli.py.

Requirements:
- Read the Java file to understand the logic — do NOT run or compile it
- Reproduce all three operations: echo, get_time, calculate
- Use argparse with subcommands: echo, get-time, calculate
- Preserve output format exactly as specified in the Javadoc:
    echo     → "Echo: {text}"
    get_time → "Current time: yyyy-MM-dd HH:mm:ss"
    calculate → "Result: {a} {operation} {b} = {result}"
- Handle the same edge cases: division by zero, unknown operation
- Must work on Windows (python) and macOS/Linux (python3)
- Add usage examples in the file header comments
```

### Step 2.2 — Review what AI produced

Before running anything, read the generated script:
- Does `echo` prepend `"Echo: "` — not `"echo: "` or anything else?
- Does `get-time` use the format `yyyy-MM-dd HH:mm:ss`?
- Does `calculate` output `"Result: 42 multiply 17 = 714"` (same as module 103)?
- Is division by zero handled with a clean error message, not a Python traceback?

If anything differs from the Java spec, ask AI to fix it with a reference to the file.

## Part 4: Create the Instruction File

The tool works. Now write the instruction — the "manual for your LLM."

### Step 4.1 — Ask AI to write the instruction

```
Create an instruction file at ./modules/104-port-to-skills/instructions/tools-cli.agent.md.

The instruction should tell an LLM:
- When to use this tool (any request for echo, current time, or arithmetic)
- How to call ./modules/104-port-to-skills/tools/tools_cli.py with each subcommand
- Exact parameter names and types for each subcommand
- What the output format looks like
- What NOT to do: never attempt to calculate manually or guess the current time — always use the script

Include one worked example for each of the three subcommands.
```

### Step 4.2 — Review the instruction

A good instruction answers: "If a user asks me to echo something / get the time / do arithmetic, what exact command do I run?"

If the instruction is vague about the `--operation` allowed values, ask AI to be explicit:

```
Update ./modules/104-port-to-skills/instructions/tools-cli.agent.md to list
the four allowed operations explicitly: add, subtract, multiply, divide.
This prevents the LLM from guessing and passing an unsupported value.
```

## Understanding Check

1. **Why is the Java service used as a reference if you never run it?**

   Expected answer: The Java service encodes the complete specification: exact output formats, edge case handling, operation names. Reading the code is more precise than reading a text description. AI can port from code without any ambiguity.

2. **The echo output in module 100 was `"Echo: Hello"`. The output of your CLI script should also be `"Echo: Hello"`. Why does this matter?**

   Expected answer: Consistency across forms means downstream code, logs, and tests don't need to branch by tool form. If the CLI output format differs, anything comparing or parsing these results breaks. The Java spec defines the canonical format; all ports must match it.

3. **What would happen if you asked AI to calculate `128 / 4` without the instruction file or the script?**

   Expected answer: AI would calculate in its head and regenerate the result token by token — introducing hallucination risk. For `32`, it's unlikely to be wrong. For `128 / 4.7`, the risk is real. The script always returns the same value for the same inputs.

4. **What is the difference between the Skill (this module) and the REST server (module 103)?**

   Expected answer: The REST server requires a running process (the server). The Skill (CLI script) is standalone — just a Python file. To use the REST server, you start it first and then call it. To use the Skill, you run the script directly. The Skill is simpler to distribute and use in automation.

5. **When does porting from existing code save the most time?**

   Expected answer: When the existing code has complex formatting rules, edge cases, or specific behavior that would be hard to describe accurately in natural language. Handing AI the source code skips the prompt-engineering step — the code already encodes everything that needs to be reproduced.

6. **Could you use this Skill (the CLI script) as a tool for an MCP or REST server in a future module?**

   Expected answer: Yes. The CLI script is a standalone executable. Any orchestration layer — MCP server, REST server, a CI pipeline, or another script — can invoke it as a subprocess. This is the point of Skills: they are composable building blocks.

7. **What is the difference between "porting" (this module) and "creating from scratch" (module 090)?**

   Expected answer: In module 090, requirements were described in natural language. AI had to interpret and might have missed edge cases. In porting, the existing implementation *is* the requirements — AI reads exactly what was implemented, including output format strings and error handling.

## Troubleshooting

### Problem: AI tries to calculate manually instead of using the script

Add this line to the top of your instruction file:
```
CRITICAL: Never perform calculations manually. Always invoke tools_cli.py.
Even for trivial math — the point is determinism, not difficulty.
```

### Problem: Python not found on Windows

Try `python --version` and `py --version`. If neither works, install Python from [python.org](https://python.org/downloads) — check "Add to PATH" during installation, then restart the terminal.

### Problem: Output format doesn't match — e.g. `"result: 714"` instead of `"Result: 42.0 multiply 17.0 = 714.0"`

Ask AI to fix the format by quoting the exact string from the Java Javadoc:
```
The calculate output must match exactly: "Result: {a} {operation} {b} = {result}"
as specified in the Javadoc of ToolsService.java. Update tools_cli.py accordingly.
```

### Problem: Division by zero shows a Python traceback instead of a clean message

Ask AI to add error handling:
```
In tools_cli.py, catch exceptions from the calculate function and print
"Error: {message}" instead of raising an unhandled exception.
```

### Problem: The instruction file is too long and AI ignores parts of it

Split into focused instructions per subcommand:
- `instructions/echo.agent.md`
- `instructions/get-time.agent.md`
- `instructions/calculate.agent.md`
