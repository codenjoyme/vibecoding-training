# Port Existing Code into Skills - Hands-on Walkthrough

In the real world, business logic already exists somewhere — in a Java Spring service, a C# class, a Python function, or an internal API. When you need a CLI tool or an AI Skill, you don't have to describe requirements from scratch. You hand the AI the existing code and say: "Port this." The AI reads the implementation, understands every edge case and rounding rule, and reproduces the same logic in your target language. No prompt engineering. No hallucinated formulas.

In this module, you'll take a provided Java Spring service and turn it into a cross-platform Python CLI tool packaged as a reusable AI Skill.

## Prerequisites

- Complete [AI Skills & Tools Creation](../090-ai-skills-tools-creation/about.md) — you must understand the Skill formula (instruction + tool)
- Complete [CLI: Command Line Interface](../103-cli-command-line-interface/about.md) — you must understand why CLI bypasses LLM for deterministic results
- Python 3 available in your terminal (`python --version` or `python3 --version`)

## What We'll Build

| Artifact | Path | Purpose |
|---|---|---|
| Reference Java service | `tools/reference/ProjectBudgetService.java` | Source of truth for business logic (already provided — do not run) |
| Python CLI tool | `tools/budget_calculator.py` | Cross-platform script generated by AI from the Java reference |
| Instruction file | `instructions/budget-calculator.agent.md` | Manual that tells LLM when and how to use the tool |
| Skill | instruction + tool together | Reusable, shareable capability for budget calculations |

The Java service is a **mock reference** — you never compile or run it. Its value is purely as a specification that AI can read and port.

## Part 1: Understand What You're Porting

Before asking AI to do anything, read the reference code yourself.

### Step 1.1 — Open the reference service

Open `./tools/reference/ProjectBudgetService.java`.

Read through it and notice:
- Three public methods: `calculateProjectCost`, `calculateROI`, `calculateBurnRate`
- Each method has Javadoc with precise formulas and edge cases
- The `round()` helper uses `HALF_UP` rounding to 2 decimal places
- Edge cases: division by zero is caught and throws `IllegalArgumentException`
- Result objects carry all intermediate values, not just the final number

**Key insight:** The Javadoc and implementation together are a complete specification. You don't need a separate requirements document — the code *is* the requirements.

### Step 1.2 — Identify what the CLI needs to support

From the Java service, you can see the CLI must support three commands:

| Command | Parameters |
|---|---|
| `project-cost` | `--team`, `--rate`, `--days`, `--overhead` |
| `roi` | `--investment`, `--profit` |
| `burn-rate` | `--budget`, `--spent`, `--elapsed`, `--total` |

You now have a complete picture. Time to delegate.

## Part 2: Ask AI to Port the Logic

Open your AI assistant with Agent Mode enabled (Claude Sonnet 4.5 recommended).

### Step 2.1 — Give AI the reference and request porting

Use this prompt exactly:

```
I have a Java Spring service at ./modules/104-port-to-skills/tools/reference/ProjectBudgetService.java.

Port its business logic to a Python CLI script at ./modules/104-port-to-skills/tools/budget_calculator.py.

Requirements:
- Read the Java file to understand the logic — do NOT run or compile it
- Reproduce all three calculations exactly: project-cost, roi, burn-rate
- Use argparse with subcommands matching the method names
- Preserve the same rounding behavior (HALF_UP to 2 decimal places)
- Handle the same edge cases (e.g., zero investment, zero elapsed days)
- Print results in a clear labeled format showing all intermediate values
- Add usage examples in the file header comments
- Must work on Windows (python), macOS, and Linux (python3)
```

AI will read the Java file and generate a Python script that mirrors the same formulas, rounding, and error handling.

### Step 2.2 — Review what AI produced

Before running anything, read the generated script:
- Does it have all three subcommands?
- Does the `project-cost` formula match? (`totalHours = team * days * 8.0`)
- Does `roi` use `(netProfit / investment) * 100`?
- Does `burn-rate` compute `dailyBurnRate = spent / elapsed` and then project forward?
- Is there error handling for zero investment and zero elapsed days?

If anything is missing, ask AI to fix it:

```
The burn-rate command is missing the overBudget flag.
Update ./modules/104-port-to-skills/tools/budget_calculator.py to add a
"⚠️ OVER BUDGET" warning when projectedTotal > budget, matching the Java service behavior.
```

## Part 4: Create the Instruction File

The tool works. Now create the instruction — the "manual for your LLM" that tells it *when* and *how* to use this tool.

### Step 4.1 — Ask AI to write the instruction

```
Create an instruction file at ./modules/104-port-to-skills/instructions/budget-calculator.agent.md.

The instruction should tell an LLM:
- When to use this tool (any request for project cost, ROI, or burn rate calculation)
- How to call ./modules/104-port-to-skills/tools/budget_calculator.py with each subcommand
- What parameters each subcommand expects (with types and examples)
- How to present results to the user (clear labeled output, highlight warnings)
- What NOT to do: never attempt manual calculation — always use the script

Include a worked example for each of the three subcommands.
```

### Step 4.2 — Review the instruction

The instruction file should answer: "If a user asks me to calculate project budget / ROI / burn rate, what exact command do I run and how do I present the result?"

If the instruction is vague, refine it:

```
Update ./modules/104-port-to-skills/instructions/budget-calculator.agent.md to be more specific
about parameter types: --team is an integer, --rate is a float in USD, --overhead is a percentage
(e.g. 20 for 20%, not 0.20). This prevents parameter format errors.
```

## Understanding Check

1. **Why is existing code a better specification than a text description of requirements?**

   Expected answer: Code is unambiguous — it contains the exact formula, rounding behavior, edge cases, and variable types. A text description requires interpretation; code requires none. The LLM can read the implementation and reproduce it exactly without guessing.

2. **Why don't we need to compile or run the Java service?**

   Expected answer: We are not using the Java service as executable software — we are using it as a *readable specification*. LLMs are very good at reading code and understanding its intent. The Java code tells us exactly what the Python script must do; we never need to execute the Java.

3. **What is the difference between "porting" (this module) and "creating from scratch" (module 090)?**

   Expected answer: In module 090, requirements were described in natural language (compound interest formula). AI had to interpret the description and might have missed edge cases. In porting, the existing implementation *is* the requirements — AI reads exactly what was implemented, including rounding rules, error handling, and intermediate values.

4. **If a user asks you to calculate a project's burn rate and you have this Skill, what should you do?**

   Expected answer: Follow the instruction file (`budget-calculator.agent.md`), invoke `budget_calculator.py burn-rate` with the correct parameters, and present the output. Never attempt to calculate manually — the script is deterministic, the LLM is not.

5. **What risk exists if the AI skips the tool and calculates manually instead?**

   Expected answer: Hallucination risk. LLMs regenerate numbers token-by-token and can silently produce wrong values (digit swaps, wrong rounding, missed edge cases). The script always produces the same result for the same inputs — the LLM does not.

6. **What makes this Skill "portable" — usable by any team member or AI assistant?**

   Expected answer: The instruction file is a self-contained manual that tells any LLM exactly what to do. The script accepts all values as parameters (no hardcoded values). Anyone with Python 3 and the two files (instruction + script) has the complete Skill, regardless of context.

7. **When does porting from existing code save the most time?**

   Expected answer: When the existing code has complex logic, multiple edge cases, or specific rounding/formatting rules. In those cases, describing requirements in natural language would take longer and risk missing details that the original code handles. The code already encodes all the requirements — porting skips the prompt engineering step entirely.

## Troubleshooting

### Problem: AI tries to calculate manually instead of using the script

Add this line to the top of your instruction file:
```
CRITICAL: Never perform calculations manually. Always invoke budget_calculator.py.
Any number that appears to be a calculation result must come from the script output, not from you.
```

### Problem: Python not found on Windows

Try `python --version` and `py --version`. If neither works, install Python from [python.org](https://python.org/downloads) — check "Add to PATH" during installation, then restart the terminal.

### Problem: Script produces different rounding than expected

The Java service uses `HALF_UP` rounding (standard financial rounding). If the generated Python script uses Python's default `round()` (which uses "banker's rounding"), ask AI to fix it:
```
Update the script to use HALF_UP rounding:
from decimal import Decimal, ROUND_HALF_UP
def round_half_up(value, decimals=2):
    d = Decimal(str(value))
    return float(d.quantize(Decimal('0.01'), rounding=ROUND_HALF_UP))
```

### Problem: AI passes parameters in wrong format (percentage as decimal)

The Java service expects `overheadPct = 20` for 20%, not `0.20`. If the script (or AI calling it) uses the decimal form, update the instruction to say explicitly:
```
--overhead is a whole number percentage (e.g. 20 for 20%, not 0.20)
```

### Problem: The instruction file is too long and AI ignores parts of it

Split into focused instructions per subcommand, following the Single Responsibility Principle from the creating-instructions module. Create:
- `instructions/project-cost.agent.md`
- `instructions/roi.agent.md`
- `instructions/burn-rate.agent.md`
