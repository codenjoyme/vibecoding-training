# Саммари сессии: Редизайн модуля 075

## Над чем работаем

Редизайн или обновление модуля **075 — Shared Instructions & Team Conventions** на основе исследования, в котором изучается более продвинутый и масштабируемый подход к управлению AI-инструкциями в командах и сервисах.

## Исходный материал

Исследование (`research.md`) описывает полностью проработанную систему с:

- **Центральным Git-репозиторием** как единственным источником истины для всех AI-инструкций ("навыков")
- **Манифест-ориентированной** моделью выбора (`_global.json` + по группам + по агентам через папку `.manifest/`)
- **Sparse checkout** — разработчики получают только нужные им навыки
- **Go CLI** (бинарник `skills`), который автоматизирует все Git-операции и создаёт PR
- **Поддержкой валидации** через `evals.json` для каждого навыка *(будет позже)*
- Полноценной **моделью совместной работы** с владением и требованиями к одобрениям

Это значительно дальше того, что есть в текущем модуле 075, который описывает более простой командный репозиторий и ручной Git-воркфлоу.

## Принятые решения

### 1. Уровень аудитории

> **Ответ:** Тренинг для всех — включая разработчиков без опыта. Не только для менеджеров. Все упоминания «менеджеры / не-инженеры» следует обновить, чтобы отразить более широкую аудиторию.
>
> Как участники без технического бэкграунда справятся с настройкой: в модуле будет `SKILL.md`, описывающий установку системы на базе CLI. AI-агент, прочитавший этот скилл, проведёт пользователя через настройку шаг за шагом — предварительных технических знаний не требуется.

### 2. Go CLI — реальный инструмент или учебная концепция?

> **Ответ:** **Реальный инструмент**, который мы создаём и публикуем как часть курса.
>
> Модуль будет содержать подпапку `tools/` с production-ready проектом Go CLI. Пользователи смогут скачать и использовать его напрямую. Инструкции по установке и использованию будут в `SKILL.md` модуля. Отдельный репозиторий не создаётся — CLI живёт внутри этого модуля.

### 3. Масштаб обновления модуля

> **Ответ:** Вариант C — **разбивка на два модуля:**
> - Модуль **075** остаётся вводным (текущий контент, лёгкое обновление) — основы и внедрение в команде
> - Модуль **076** становится продвинутой системой (папка `.manifest/`, sparse checkout, CLI `skills`, управление)

### 4. `evals.json` — включать или нет?

> **Ответ:** Не в этом модуле. Пометить как **«coming soon»** в `SKILL.md` и в research-документе. Тестирование и валидация инструкций — отдельная тема на потом.

### 5. IDE-специфичные обёртки

> **Ответ:** Рекомендуем **tools-agnostic подход** из `creating-instructions.agent.md`: все инструкции хранятся как plain Markdown файлы в отдельной папке, независимо от агентной системы или IDE. Каждая IDE/агент имеет свои тонкие адаптеры-обёртки (ссылки) на эти файлы. Это исключает привязку к конкретному вендору и делает содержимое навыков портируемым между IDE (VSCode Copilot, Cursor, Claude Code и т.д.).

## План работы

На основе принятых решений:

1. **Формулировка проблемы** — Почему индивидуальные инструкции не масштабируются на уровне команды
2. **Центральный репозиторий навыков** — Структура, анатомия навыка (`SKILL.md` + `README.md`)
3. **Паттерн манифеста** — `_global.json`, `_agents.json`, `<имя-группы>.json` через папку `.manifest/`; саб-конфиги для тематических групп
4. **Воркфлоу sparse checkout** — Подтягивать только то, что нужно (`skills init --groups ...`)
5. **Воркфлоу участия** — Ветка → PR → ревью владельца → мердж → все получают обновление
6. **Владение и управление** — Кто владеет глобальными навыками, требования к одобрениям (рекомендательно)
7. **CLI `skills`** — Реальный инструмент в папке `tools/`; `SKILL.md` сопровождает установку
8. **evals** — Coming soon, упоминается, но не преподаётся

Текущий контент модуля 075 (паттерн COIN, правила разрешения конфликтов, структура папок) соответствует пунктам 1, 5 и 6 — переиспользуемый материал.
