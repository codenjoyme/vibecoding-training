# Система управления инструкциями
## Git + Sparse Checkout + Go CLI

---

## Обзор

Этот документ описывает лёгкую, масштабируемую систему управления AI-инструкциями (промптами) в условиях множества микросервисов и команд. Система построена на Git как источнике истины, sparse checkout для частичного использования рабочего пространства и кроссплатформенном Go CLI, распространяемом в виде единого бинарного файла.

Система разработана для поддержки:

- Сред с несколькими репозиториями
- Межкомандного взаимодействия
- Переиспользуемых и компонуемых единиц инструкций ("навыков")
- Минимальной инфраструктуры (без тяжёлых LLMOps-платформ)

---

## Основные принципы

| Принцип | Описание |
|---|---|
| **Git — единственный источник истины** | Все инструкции хранятся в одном центральном Git-репозитории. |
| **Инструкции модульны** | Каждая инструкция — самодостаточный "навык", хранящийся в собственной директории. |
| **Без дублирования** | Каждый навык существует только один раз и может переиспользоваться в разных сервисах. |
| **Выбор через манифест** | Глобальный манифест определяет, какие навыки используются какими сервисами или агентами. |
| **Частичное рабочее пространство через sparse checkout** | Разработчики подтягивают только нужное им подмножество навыков. |
| **Абстракция через CLI** | Go CLI скрывает операции Git, sparse checkout и PR-воркфлоу. |

---

## Структура репозитория

Все инструкции хранятся в едином плоском репозитории, где каждый навык — это директория:

```
.instructions/repo/
├── .manifest/
│   ├── global.json
│   ├── agents.json
│   ├── service1.json
│   ├── service2.json
│   └── security.json
├── code-review-base/
│   ├── skill.md
│   ├── evals.json
│   └── README.md
├── code-review-gpt/
│   ├── skill.md
│   ├── evals.json
│   └── README.md
├── security-guidelines/
│   ├── skill.md
│   └── README.md
├── style-guidelines/
│   ├── skill.md
│   └── README.md
└── agent-copilot/
    ├── skill.md
    ├── evals.json
    └── README.md
```

**Ключевые правила:**

- Каждая директория = один навык
- `skill.md` = основное содержимое инструкции
- `evals.json` = необязательные тест-кейсы
- `README.md` = метаданные (владелец, описание)
- Папка `.manifest/` содержит все конфигурационные файлы (см. Конфигурация манифеста ниже)
- Вложенная иерархия не требуется (плоская структура)

---

## Конфигурация манифеста

Вместо единственного `manifest.json` система использует папку `.manifest/`, содержащую отдельные конфигурационные файлы. Это устраняет конфликты слияния, когда несколько команд одновременно добавляют свои проекты.

### Структура файлов

| Файл | Назначение |
|---|---|
| `global.json` | Навыки, применяемые ко всем рабочим пространствам по умолчанию |
| `agents.json` | Привязки навыков к конкретным IDE/инструментам (VSCode, Copilot, Cursor и т.д.) |
| `<имя-проекта>.json` | Выбор навыков для конкретного проекта или сервиса |
| `security.json` *(пример саб-конфига)* | Общая тематическая группа, переиспользуемая между проектами |

### `global.json`

```json
{
  "skills": ["style-guidelines"]
}
```

### `agents.json`

```json
{
  "copilot": ["agent-copilot"],
  "cursor": [],
  "vscode": []
}
```

### `<имя-проекта>.json` (например, `service1.json`)

```json
{
  "skills": ["code-review-base", "security-guidelines"],
  "sub-configs": ["security"]
}
```

### Саб-конфиг (например, `security.json`)

```json
{
  "skills": ["security-guidelines", "owasp-top10"]
}
```

**Правила:**

- Навыки из `global.json` всегда включены для всех
- `agents.json` определяет навыки для конкретных IDE/инструментов
- Каждый файл проекта перечисляет нужные навыки и опционально ссылается на саб-конфиги
- Саб-конфиги — это переиспользуемые между проектами группы навыков (например, `security.json`, `testing.json`)
- Имя файла = имя проекта или имя саб-конфига
- Навыки указываются по имени директории

## Локальный воркфлоу

Все операции выполняются через CLI.

### Инициализация

```bash
instructions init --project <имя-проекта> [<имя-проекта-2> ...]
```

**Поведение:**
1. Клонировать центральный репозиторий в `.instructions/repo`
2. Прочитать `.manifest/<имя-проекта>.json` для каждого указанного проекта
3. Определить нужные навыки: `global.json` + навыки проекта + указанные саб-конфиги + `agents.json` (для текущей IDE)
4. Выполнить sparse checkout для всех найденных директорий навыков

Несколько проектов в одном рабочем пространстве:

```bash
instructions init --project service1 service2
```

**Результат:** Локально присутствуют только релевантные навыки.

### Работа с навыками

Разработчики редактируют файлы напрямую:

```
.instructions/repo/<имя-навыка>/skill.md
```

Опционально также обновляют:
- `evals.json`
- `README.md`

### Публикация изменений

```bash
instructions push <имя-навыка>
```

**Поведение:**
1. Создать новую ветку: `feature/<имя-навыка>-update`
2. Добавить в индекс только изменённые файлы
3. Закоммитить изменения
4. Запушить в удалённый репозиторий
5. Автоматически создать Pull Request
6. Вернуть URL пул-реквеста

### Получение обновлений

```bash
instructions pull
```

**Поведение:**
- `git pull` на репозитории
- Повторное применение sparse checkout при необходимости

## Стратегия Sparse Checkout

Sparse checkout используется для ограничения локальных файлов только необходимым.

CLI внутренне выполняет:

```bash
git sparse-checkout init --cone
git sparse-checkout set <навык-1> <навык-2> ...
```

Навыки определяются из манифеста автоматически.

**Преимущества:**
- Минимальный локальный след
- Отсутствие дублирования
- Полная совместимость с Git
- Чистые диффы и коммиты

---

## Дизайн навыка

Каждая директория навыка содержит:

### `skill.md`

Основное содержимое инструкции — написано в plain Markdown, без привязки к IDE.

### `evals.json` *(необязательно)*

Тест-кейсы для автоматической валидации:

```json
[
  {
    "input": "пример кода",
    "expected": "должен упомянуть обработку ошибок"
  }
]
```

### `README.md`

Метаданные навыка:

- **Описание** — что делает этот навык
- **Владелец** — кто несёт ответственность
- **Контекст использования** — когда применять

---

## Компоновка промпта

Финальный промпт конструируется динамически путём конкатенации:

```
final_prompt = глобальные навыки + навыки сервиса + навыки для конкретной модели (опционально)
```

Варианты для конкретных моделей могут быть реализованы как отдельные навыки:

| Навык | Целевая модель |
|---|---|
| `code-review-base` | Любая модель |
| `code-review-gpt` | Оптимизировано для GPT-4 |
| `code-review-claude` | Оптимизировано для Claude |

---

## Версионирование

Версионирование обеспечивается полностью через Git:

- **Хэш коммита** = идентификатор версии
- **Ветки** для разработки и ревью
- **Теги** для релизов *(необязательно)*

---

## Модель совместной работы

- Каждый навык имеет **владельца** (указан в `README.md`)
- **PR обязателен** для всех изменений — прямые коммиты в main запрещены
- Минимум **1 одобрение** от владельца навыка перед мерджем
- **Глобальные навыки требуют 2 одобрений** (они влияют на всех)

---

## Управление конфликтами

Лучшие практики для минимизации конфликтов:

- Небольшие, сфокусированные PR
- Одно логическое изменение на PR
- Избегать редактирования нескольких навыков в одном PR
- Предпочтительнее разбивать большие промпты на несколько навыков

## Архитектура Go CLI

### Обзор

CLI реализован на Go и распространяется как единый бинарный файл.

**Ключевые свойства:**

- Кроссплатформенность (Windows, macOS, Linux)
- Отсутствие зависимостей от среды выполнения
- Единый исполняемый файл

### Команды CLI

| Команда | Описание |
|---|---|
| `instructions init --project <имя> [<имя-2> ...]` | Инициализировать рабочее пространство с sparse checkout для одного или нескольких проектов |
| `instructions pull` | Получить обновления с удалённого репозитория |
| `instructions push <имя-навыка>` | Запушить изменения и создать PR |
| `instructions list` | Показать список доступных навыков |
| `instructions eval <имя-навыка>` | Запустить evals для навыка против LLM |

### Внутренние модули (Go)

| Модуль | Ответственность |
|---|---|
| **manifest loader** | Чтение файлов `.manifest/<имя-проекта>.json`, объединение с `global.json` и `agents.json`, разрешение зависимостей навыков |
| **git manager** | Клонирование репо, обновление, создание веток, коммит и пуш |
| **sparse manager** | Настройка sparse checkout, обновление выбранных навыков |
| **PR integration** | Создание пул-реквестов через API Git-провайдера |
| **eval runner** | Запуск eval-кейсов против LLM |

---

## Дистрибуция

CLI компилируется в платформо-зависимые бинарники:

| Платформа | Бинарник |
|---|---|
| Windows | `instructions.exe` |
| Linux / macOS | `instructions` |

**Методы дистрибуции:**

- GitHub Releases (публичные или приватные)
- Внутреннее хранилище артефактов / пакетный реестр

---

## Ограничения

Данная система **НЕ** обеспечивает:

- Обновления в реальном времени (инструкции загружаются при старте)
- Встроенную инфраструктуру A/B-тестирования
- Продвинутые дашборды наблюдаемости

Всё это можно добавить позже в виде расширений при необходимости.

Данная система обеспечивает:

|---|---|
| Централизованное управление инструкциями | Единый Git-репозиторий |
