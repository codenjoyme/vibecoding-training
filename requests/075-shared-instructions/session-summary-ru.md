# Саммари сессии: Редизайн модуля 075

## Над чем работаем

Редизайн или обновление модуля **075 — Shared Instructions & Team Conventions** на основе исследования, в котором изучается более продвинутый и масштабируемый подход к управлению AI-инструкциями в командах и сервисах.

## Исходный материал

Исследование (`research.md`) описывает полностью проработанную систему с:

- **Центральным Git-репозиторием** как единственным источником истины для всех AI-инструкций ("навыков")
- **Манифест-ориентированной** моделью выбора (глобальные + для конкретного сервиса + для конкретного агента)
- **Sparse checkout** — разработчики получают только нужные им навыки
- **Go CLI** (бинарник `instructions`), который автоматизирует все Git-операции и создаёт PR
- **Поддержкой валидации** через `evals.json` для каждого навыка
- Полноценной **моделью совместной работы** с владением и требованиями к одобрениям

Это значительно дальше того, что есть в текущем модуле 075, который описывает более простой командный репозиторий и ручной Git-воркфлоу.

## Уточняющие вопросы перед началом работы

Прежде чем трогать модуль, мне нужны твои решения по следующим пунктам:

### 1. Уровень аудитории

> Текущий модуль ориентирован на менеджеров / не-инженеров, осваивающих AI-assisted разработку. Система из исследования (Go CLI, sparse checkout, manifest JSON) — это **территория разработчиков/DevOps**.
>
> **Вопрос:** Мы редизайним это для технической аудитории, или нужно сохранить доступность для не-инженеров? Или разбить на два уровня?

### 2. Go CLI — реальный инструмент или учебная концепция?

> Исследование описывает Go CLI как механизм дистрибуции. Его создание — это полноценный инженерный спринт, а не учебное задание.
>
> **Вопрос:** CLI должен быть:
> - (a) **Реальным инструментом**, который мы создаём и публикуем как часть курса?
> - (b) **Концептуальным учебным примером** — "вот как выглядит зрелая система"?
> - (c) **Вне скоупа** — сосредоточиться на паттерне Git + манифест, преподавать концепцию без создания бинарника?

### 3. Масштаб обновления модуля

> Сейчас модуль 075 рассчитан на 15 минут и охватывает базовую структуру общего репозитория + Git-воркфлоу.
>
> **Вопрос:** Цель:
> - (a) **Полностью заменить** текущий модуль новой системой?
> - (b) **Расширить** — сохранить текущий контент как основу, добавить продвинутую систему отдельной "Частью 2" или новым модулем (например, 076)?
> - (c) **Оставить текущий модуль** как есть, но добавить ссылку/приложение с продвинутым подходом?

### 4. `evals.json` — включать или нет?

> Исследование включает фреймворк для валидации инструкций (тест-кейсы, запускаемые против LLM).
>
> **Вопрос:** Тестирование и валидация инструкций — это то, чему хотим учить в этом модуле, или отдельная тема на потом?

### 5. IDE-специфичные обёртки

> Ядро системы из исследования не привязано к IDE. Текущий модуль 075 упоминает совместимость с VSCode и Cursor, но не углубляется.
>
> **Вопрос:** Стоит ли включить раздел о том, как центральные `skill.md`-файлы соотносятся с IDE-специфичными форматами обёрток (`.github/prompts/`, `.cursor/rules/`)?

## Предварительный план (в ожидании твоих ответов)

На основе исследования вероятная структура обновлённого/нового модуля:

1. **Формулировка проблемы** — Почему произвольные индивидуальные инструкции не масштабируются на уровне команды
2. **Центральный репозиторий навыков** — Структура, анатомия навыка (`skill.md` + `README.md`)
3. **Паттерн манифеста** — Глобальные vs. сервисно-специфичные vs. агентно-специфичные навыки
4. **Воркфлоу sparse checkout** — Подтягивать только то, что нужно
5. **Воркфлоу участия** — Ветка → PR → ревью владельца → мердж → все получают обновление
6. **Владение и управление** — Кто владеет глобальными навыками, требования к одобрениям
7. **Концепция CLI** — Абстрагировать сложность Git (создавать или концепция — TBD)
8. **evals** — Опционально: как тестировать качество инструкций

Контент текущего модуля (паттерн COIN, правила разрешения конфликтов, структура папок) частично соответствует пунктам 1, 5 и 6 — то есть есть материал для переиспользования.
