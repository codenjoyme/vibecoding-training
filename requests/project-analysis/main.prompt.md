## UPD2

Проанализируй слабые места подхода, используемого в этом проекте.

Выводы из анализа:

1. **Хрупкость из-за недетерминированности AI** — весь флоу зависит от того, правильно ли AI прочитает инструкцию. В `main.agent.md` есть раздел "CRITICAL FILE LOADING RULES" — признание системной проблемы с частичным чтением файлов.

2. **Нет автоматической проверки выполнения** — нет тестов, что AI выполнил инструкцию правильно. Прогресс в `training-progress.md` обновляет сам AI, который может ошибиться.

3. **Высокая нагрузка на контекст** — каждый промпт тянет за собой `main.agent.md` и другие файлы. При 30+ модулях каталог разрастается, растут затраты токенов.

4. **Тяжёлое сопровождение** — если меняется соглашение, нужно обновить десятки файлов. Нет механизма проверить консистентность структуры модулей.

5. **Зависимость от конкретной модели** — качество обучения прямо зависит от модели. "Tool-agnostic" — маркетинговое, но не техническое утверждение.

6. **Нет чёткой границы между данными и кодом** — инструкции это одновременно "программа" (логика) и "данные" (контент). Работает на старте, но не масштабируется.

7. **Сложность входа для новых авторов** — несмотря на UX "просто поговори с AI", автору нужно знать систему папок, соглашения, структуру модулей и несколько instruction-файлов.

**Короткий вывод:** подход отлично работает для прототипирования и небольших команд, но плохо масштабируется и имеет слабые гарантии корректности выполнения.

### RESULT

Анализ слабых мест задокументирован в этом файле. Выявлено 7 структурных проблем подхода "инструкции для AI как основа проекта".

## UPD4

Хорошо, как мне улучшить подход чтобы закрыть те проблемы, что есть? go

### RESULT

Конкретные улучшения по каждой нерешённой проблеме:

---

**1. Хрупкость / недетерминированность AI**

→ Добавить `linting` инструкцию: скрипт PowerShell/Python который проверяет что все `about.md` имеют обязательные поля (`# About`, duration, skill, topics). Запускать как pre-commit hook или вручную. AI не нужен — детерминированная проверка.

→ В критических инструкциях добавить "контрольную сумму" — список финальных шагов который агент должен подтвердить (check-list в конце `### RESULT`).

---

**2. Нет автоматической проверки выполнения**

→ Уже частично решается `### RESULT`. Дополнение: добавить в `training-progress.md` раздел `## Verification` где для каждого пройденного модуля агент фиксирует конкретный артефакт (файл, команду, скриншот) — не только текстовый фидбек.

→ Для модулей с кодом: добавить `tools/check.ps1` / `tools/check.sh` — скрипт, который агент запускает и фиксирует вывод в `RESULT`.

---

**3. Высокая нагрузка на контекст**

→ Разбить `main.agent.md` на два слоя: `main.agent.md` (только диспетчер, <50 строк) + отдельные файлы инструкций загружаются лениво по ключевым словам. Это уже частично сделано через `<follow>` теги — расширить этот паттерн.

→ Использовать `iterative-prompt` подход для самого обучения (`training-mode-iterative-prompt.agent.md` уже существует) — контекст сессии в файле, а не в памяти модели.

---

**4. Тяжёлое сопровождение**

→ Создать `scripts/validate-modules.ps1` — скрипт обходит все папки в `modules/`, проверяет наличие `about.md` и `walkthrough.md`, выводит список нарушений. Запускать при каждом создании модуля.

→ Шаблоны (`about.md.template`, `walkthrough.md.template`) уже есть в `create-training-module.agent.md` — вынести их в отдельные файлы `modules/_template/`, чтобы при изменении соглашения обновлять одно место.

---

**5. Зависимость от конкретной модели**

→ В `about.md` каждого модуля добавить поле `tested-with: Claude Sonnet 4.6` — минимальная документация совместимости.

→ Критические инструкции писать максимально императивно (numbered steps, no ambiguity) — это уменьшает разброс между моделями. Проверить топ-5 инструкций на "императивность".

---

**6. Нет границы данные/код**

→ Разделить: `instructions/` = логика (не трогать без теста), `modules/` = данные (добавлять свободно). Добавить `CONTRIBUTING.md` с этим правилом явно.

→ Для автоматизации — вынести скрипты из инструкций в `tools/` папки с чистым CLI интерфейсом. AI-инструкция вызывает скрипт, не содержит его код.

---

**7. Сложность входа для авторов**

→ Создать `modules/_template/` с готовой заглушкой + `modules/_template/README.md` на 10 строк: "скопируй, переименуй, заполни поля". Ссылку из `for-authors.md`.

→ Добавить в `requests/` папку `_examples/` с минимальным примером iterative-prompt цикла (3 UPD + RESULT). Сейчас примеры есть, но не выделены как "стартовые".

---

**Приоритет по усилию/эффекту:**
1. `scripts/validate-modules.ps1` — низкое усилие, высокий эффект на сопровождение
2. Verification артефакты в `training-progress.md` — низкое усилие, повышает доверие к прогрессу
