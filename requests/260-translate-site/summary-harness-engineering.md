# Саммари: Harness Engineering — Leveraging Codex in an Agent-First World

**Автор:** Ryan Lopopolo (OpenAI) | **Дата:** 11 февраля 2026 | **Источник:** [openai.com](https://openai.com/index/harness-engineering/)

## Главная идея

Команда OpenAI за 5 месяцев построила внутренний продукт (~1 млн строк кода, 1500 PR) с **0 строк кода, написанных вручную**. Всё — логика, тесты, CI, документация, инструментарий — написано Codex. Оценка: в 10 раз быстрее ручной разработки.

**Мантра: «Люди направляют. Агенты исполняют.»**

## Ключевые уроки

### 1. Роль инженера изменилась
- Инженер больше не пишет код — он **проектирует среды, формулирует намерения и строит петли обратной связи**
- Когда что-то ломается: «какой возможности не хватает агенту?» вместо «попробуй сильнее»
- 3.5 PR на инженера в день, пропускная способность **растёт** с ростом команды (3→7 чел.)

### 2. AGENTS.md — оглавление, не энциклопедия
- Один большой AGENTS.md провалился: вытесняет контекст, устаревает мгновенно, всё «важное» = ничего важного
- Решение: ~100 строк AGENTS.md как карта + структурированная `docs/` директория — единый источник истины
- **Прогрессивное раскрытие**: агент начинает с малой точки входа и знает, куда смотреть дальше

### 3. Читаемость для агентов (Agent Legibility)
- Всё, что агент не видит в контексте, **не существует** — Slack-обсуждения, Google Docs, головы людей
- Решение: **весь контекст — в репозиторий** (версионирован, обнаруживаем)
- Codex имеет доступ к Chrome DevTools Protocol (DOM-снимки, скриншоты, навигация)
- Логи через LogQL, метрики через PromQL — промпты вроде «запуск < 800мс» становятся выполнимыми
- Один запуск Codex может работать **6+ часов** (пока люди спят)

### 4. Архитектура и вкус
- Жёсткая архитектурная модель: Types → Config → Repo → Service → Runtime → UI
- Зависимости только «вперёд», сквозные задачи через единый интерфейс Providers
- Кастомные линтеры (сгенерированные Codex) с сообщениями об ошибках = инструкции по исправлению
- **Обеспечивай инварианты, не микроменеджь реализации**
- «Скучные» технологии лучше для агентов (компонуемость, стабильность API, представленность в обучающей выборке)
- Иногда дешевле переимплементировать, чем работать с непрозрачной библиотекой

### 5. Философия мержей
- Минимум блокирующих gate'ов, короткоживущие PR
- Flake-тесты часто решаются повторным запуском, а не блокировкой
- «Это было бы безответственно в low-throughput среде. Здесь — часто правильный компромисс.»

### 6. Полная автономность (что агент делает по одному промпту)
- Проверить кодовую базу → воспроизвести баг → записать видео → исправить → записать видео с исправлением → открыть PR → ответить на ревью → исправить сборку → эскалировать только при необходимости → смержить

### 7. Энтропия и «сборка мусора»
- Codex реплицирует существующие паттерны, даже плохие → дрейф неизбежен
- Раньше: 20% времени (пятницы) на уборку «ИИ-слопа» — не масштабировалось
- Теперь: «золотые принципы» в репозитории + фоновые агенты сканируют отклонения и открывают рефакторинговые PR
- **Технический долг = кредит с высокой ставкой**: лучше платить непрерывно малыми порциями

## Что ещё не знают
- Как архитектурная когерентность будет эволюционировать **годами**
- Где человеческое суждение даёт наибольший рычаг и как его кодировать
- Как система будет эволюционировать с ростом мощности моделей
