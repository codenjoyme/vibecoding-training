# Саммари: The Dark Factory Pattern (HackerNoon)

**Автор:** Saumya Tyagi | **Дата:** 19 февраля 2026 | **Источник:** [hackernoon.com](https://hackernoon.com/the-dark-factory-pattern-moving-from-ai-assisted-to-fully-autonomous-coding)

## Главная идея

«Тёмная фабрика» — это производственный термин (завод, работающий без людей с выключенным светом). В контексте разработки ПО: **пайплайн, где ни один человек не пишет, не ревьюит и не тестирует код вручную**. Люди пишут спецификации и критерии приёмки — система делает всё остальное.

## Проблема

Команды внедрили ИИ-помощники (Copilot, Cursor, Cody), но автоматизировали только **набор текста**. Узкое место переместилось: ожидание код-ревью (2-8 часов), ручное тестирование (30-60 мин), обсуждения стиля (30-90 мин). Скорость написания кода выросла, но всё вокруг осталось тем же медленным ручным процессом.

## 4 фазы (каждая ценна сама по себе)

### Фаза 1 — Лучший контекст для агентов
- **AGENTS.md файл** в каждом репозитории (~100 строк): структура, паттерны, зависимости
- **Build-Before-Push**: агент собирает и тестирует перед пушем
- **Линтеры с инструкциями**: сообщения об ошибках как инструкции, а не описания

### Фаза 2 — Spec-Driven Development + Holdout-сценарии
- Инженер пишет **спецификацию** (markdown + YAML frontmatter)
- Кодирующий агент генерирует код **только по спецификации**
- **Holdout-сценарии** (BDD на английском) — в директории, **недоступной агенту**
- Отдельный LLM-оценщик проверяет результат (3 запуска, порог 2/3, общий порог 90%)
- **Ключевой принцип**: train/test separation из ML — агент не видит тесты

### Фаза 3 — Убираем человеческий барьер
- Авто-мерж для сервисов с хорошими метриками (>90% pass rate, <5% false positive)
- Агенты поддержки качества — еженедельная очистка от дрейфа кода

### Фаза 4 — Полная тёмная фабрика
- Авто-мерж для всех сервисов
- Тикеты с тегом `bot:fix` автоматически генерируют спецификации
- **Цифровые двойники** — мок-серверы для внешних зависимостей

## Ключевые советы

1. **Сегодня**: напишите `AGENTS.md` для самого активного репозитория (2 часа, постоянная отдача)
2. **На этой неделе**: напишите 5 holdout-сценариев для одного сервиса и проверьте LLM-оценщик
3. **Не делайте авто-мерж** пока нет 20-30 PR с совпадением оценок
4. **Не привязывайтесь к одному ИИ-провайдеру** — агент абстрагирован, замена за 60 секунд
5. **Ошибки линтера пишите как инструкции**, а не описания
6. **Поговорите с командой** — «автоматизируем скучное» должно быть правдой, а не лозунгом

## Математика

8 инженеров → производительность 25-30 инженеров при работающей тёмной фабрике. Стоимость ~\/день на инженера-эквивалент (дешевле зарплаты).

## Риски

- LLM-оценщик может одобрить плохой код (митигация: тройной запуск + существующий CI/CD)
- Разработчики не хотят перестать писать код (поэтапный подход помогает)
- Сценарии устаревают (на практике меньшая проблема — нет step-definitions, LLM адаптируется)
