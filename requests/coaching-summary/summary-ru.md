# GenAI Coaching: Ключевые темы сессии

> Краткое саммари коучинг-сессий по генеративному AI для специалистов, работающих с GitHub Copilot и языковыми моделями.

---

## 1. Как языковые модели генерируют текст

Модель не "думает" — она **предсказывает следующее слово** на основе всего текста, который видит. Как T9 в телефоне, но вместо 5 предыдущих слов она видит 200 000 слов одновременно и чувствует все связи между ними.

**Почему важно:** Понимание механизма снимает магию и позволяет осознанно управлять результатом.

## 2. Контекстное окно — "стена с газетами"

Представьте огромный экран, оклеенный газетами. Модель одним взглядом читает все статьи, видит связи между ними и генерирует связный ответ. Это и есть контекстное окно — весь текст, который модель видит одномоментно.

**Почему важно:** Всё, что вы пишете, всё, что модель отвечает, все вызовы инструментов — всё это накапливается в одном контексте и влияет на каждое следующее слово.

## 3. У модели нет памяти — она "рыбка Дори"

Закрыли сессию — модель забыла всё. Новый чат = чистый лист. Единственная "память" — файлы проекта и инструкции, которые вы подготовили заранее.

**Почему важно:** Чтобы знания сохранялись между сессиями, нужно **активно выгружать** полезные результаты в файлы-инструкции. Это ваш способ создать "долгосрочную память" для модели.

## 4. Не спорьте — редактируйте промт

Если модель дала не тот результат — **не спускайтесь ниже**, не спорьте. Вернитесь к исходному промту, добавьте недостающие факты, уточните формулировки. Каждый спор загрязняет контекст ложными утверждениями, и модель начинает галлюцинировать всё больше.

**Почему важно:** Чистый контекст = точный ответ. Загрязнённый контекст = каша из конфликтующих требований.

## 5. Температура и недетерминизм

Модели специально добавлена "температура" — элемент случайности при выборе следующего слова. Поэтому на один и тот же запрос ответы будут разными по форме, хотя схожими по смыслу.

**Почему важно:** Нельзя ожидать 100% воспроизводимости. Но можно **сузить диапазон** ответов, добавляя в промт конкретные стейтменты, пока результат не станет стабильным.

## 6. Инструкции — ваш "knowledge transfer"

Новая сессия — это джуниор, который ничего не знает о проекте. Инструкция — это knowledge transfer для этого джуниора. Через 10–20 итераций исправлений инструкция "обрастает" корнер-кейсами и превращается в синьорного агента.

**Почему важно:** Инструкции = масштабируемый опыт. Один раз описали, дальше делегируете.

## 7. Инструкция → Скилл → Агент

- **Инструкция** — текстовое описание "как сделать одно приседание"
- **Скилл** — инструкция + код (например, Python-калькулятор для точных вычислений)
- **Агент** — набор инструкций, связанных процессом: "возьми отсюда, обработай этим, положи туда"

**Почему важно:** Разделение на уровни позволяет комбинировать инструкции между собой и создавать сложные автоматизации.

## 8. Под капотом: системный промт, тулы, агентный режим

Перед вашим первым словом модель уже видит: описания 50+ инструментов, системный промт с правилами поведения, структуру проекта, пользовательские инструкции. Ваш вопрос — лишь малая часть всего контекста.

**Почему важно:** Понимание того, что происходит "за кулисами", убирает ощущение магии и даёт контроль над поведением модели.

## 9. Prompt Engineering: выход за пределы своих знаний

Если вы не знаете нужный термин — попросите модель дать 10 ключевых терминов по теме. Это "раскрывает" нужный раздел её "библиотеки" знаний. Дальше работаете в обогащённом контексте.

**Почему важно:** Вы не обязаны знать всё. Модель — это библиотека без каталога. Ваша задача — подвести её к нужной "полке".

## 10. Итеративное улучшение через галлюцинации

Галлюцинация — это не провал, а сигнал. Увидели ошибку — скажите модели: "исправь инструкцию, чтобы этого больше не было". Каждый корнер-кейс делает инструкцию надёжнее.

**Почему важно:** Галлюцинации — это обратная связь. Используйте их для эволюции инструкций, а не как повод отказаться от инструмента.

## 11. Version Control: Git и методология "baby steps"

AI часто "улучшает" рабочий код — и ломает его. Методология baby steps: после каждого успешного изменения делайте `git add`. Один коммит = одна мысль, которую можно объяснить одним предложением. Три фичи по 15 минут через baby steps — 45 минут. Те же три фичи одновременно — 2–3 часа из-за жонглирования контекстом.

**Почему важно:** Git превращает разработку с AI из "надеюсь, ничего не сломается" в "могу откатить за секунду". Это ваша страховочная сетка для работы с генеративным AI.

## 12. Управление скиллами на уровне команды

Когда индивидуальных инструкций становится 50+, их невозможно копировать вручную между проектами. Решение — центральный Git-репозиторий с инструкциями, манифесты для каждой команды и CLI-утилита `skills`, которая подтягивает в проект только нужные скиллы через sparse checkout.

**Почему важно:** Один человек исправил инструкцию — все в команде получили обновление через `skills pull`. Это переход от "каждый пишет свои правила" к единой практике по всей организации.

## 13. AI Skills & Tools: инструкция + код = надёжность

Модель **плохо считает** — она "видит сон" о математике, а не вычисляет. Решение: не просите AI считать — попросите его **создать калькулятор**. Инструкция говорит "когда нужно вычислить — запусти этот скрипт", а скрипт выдаёт точный ответ. Формула: инструкция + инструмент = скилл.

**Почему важно:** Скиллы превращают AI из ненадёжного "угадывателя" в точный и воспроизводимый инструмент. Финансовые расчёты, обработка данных, форматирование — всё, где нужна точность.

## 14. MCP — Model Context Protocol

MCP — это "USB для AI". До USB каждое устройство требовало свой драйвер. MCP стандартизирует подключение любых внешних сервисов: GitHub, базы данных, файловые системы, Jira, Confluence. Настроили один раз — и AI-агент "видит" внешний мир через единый протокол.

**Почему важно:** MCP снимает барьер "AI не имеет доступа к нашим данным". Более 250 готовых серверов уже доступны. Но помните: данные проходят через сервер того, кто создал коннектор — безопасность требует осознанности.

## 15. CLI — прямой доступ без галлюцинаций

MCP передаёт результат инструмента через модель — и она может "перегенерировать" ответ, переставив цифры. CLI (curl к REST API) возвращает результат напрямую, без AI в цепочке. Сервер вернул "714" — вы получите "714", а не "741".

**Почему важно:** Критические операции — через CLI (гарантированная точность). Исследовательские задачи, где нужен "мозг" — через MCP. Это как разделение на ручное и автоматическое управление: оба нужны, но для разных ситуаций.

## 16. MCP GitHub — делегирование задач автономным агентам

Вместо переключения между IDE и браузером — всё через чат: создание репозитория, push кода, управление issues. Ключевой паттерн — **делегирование автономному Coding Agent**: AI-сессия проводит интервью по требованиям, создаёт GitHub issue со всем контекстом, а дальше вы назначаете issue на Coding Agent — и он **реализует задачу в фоне**, пока вы заняты другим в IDE. Агент создаёт branch, пишет код, запускает тесты и открывает PR. Вы ревьюите результат, а не пишете код.

**Почему важно:** Это переход от "AI как помощник" к "AI как автономный исполнитель". GitHub issue — контракт между вами и агентом. Вы планируете — агент реализует параллельно.

## 17. SpecKit — от идеи к прототипу через спецификацию

SpecKit заставляет описать ЧТО и ЗАЧЕМ, прежде чем писать КАК. Восемь этапов: specify → clarify → plan → tasks → analyze → implement → checklist. AI **отказывается** кодить, пока спецификация не завершена. Результат — полная документация решения ещё до первой строки кода.

**Почему важно:** Ошибки на этапе спецификации стоят минуты. Ошибки на этапе реализации — дни. SpecKit масштабируется от хакатонных прототипов до продакшн-кодбейзов, потому что папка `specs/` — это живая документация.

## 18. Chrome DevTools MCP — AI тестирует ваше приложение

AI-агент получает "глаза и руки" в браузере: кликает кнопки, заполняет формы, делает скриншоты, читает консоль ошибок. В связке с hot reload возникает цикл самокоррекции: код → тест → баг → фикс → проверка — всё без участия человека за 2–3 минуты.

**Почему важно:** QA сдвигается влево — баги ловятся во время разработки, а не после релиза. Сгенерированные тест-сценарии в markdown становятся регрессионной документацией, которую можно запускать снова и снова.
