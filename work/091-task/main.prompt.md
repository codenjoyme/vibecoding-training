## 📖 Part 1: Понимаем концепцию

### Что такое Approval Testing?

Обычное тестирование: ты пишешь assertion — `expect(output).toBe("hello")`. Ты заранее знаешь что должно выйти.

**Approval Testing** (придумал Llewellyn Falco): ты запускаешь систему, захватываешь весь её вывод — и *одобряешь* его как "истину". При следующем запуске любое отличие от одобренного вывода — потенциальная регрессия.

### Три ноу-хау этого инструмента

1. **Нет автоматических assertions.** Вывод сохраняется как Markdown-снимок. Ты (или LLM, который читает `git diff`) решаешь — нормальное ли это изменение. В эпоху LLM модель может посмотреть на снимок + изменения в коде и поставить диагноз гораздо эффективнее, чем assertion-failure message.

2. **Формат на основе Markdown.** Тест — это читаемый документ с описанием *зачем* каждая команда, вперемешку с самими командами и выводом. Документация + тест + доказательство — в одном файле.

3. **Docker = OS-независимость.** Всё внутри контейнера. Windows, macOS, Linux — результаты одинаковые. Конец "у меня работало" 🙂

### Зачем это legacy-проектам?

Legacy-проект — тот, что работает, но тестов нет. Добавить unit-тесты без понимания кода сложно. Но ты *можешь* захватить что CLI делает сегодня, одобрить это как правду, и обнаружить когда будущие изменения нарушат это. Это страховочная сетка.

Знаменитая фраза: *"legacy code is code without tests"* (Майкл Физерс, автор книги Working Effectively with Legacy Code). Не "старый" — а *без тестов*. Ты буквально описал классику.

И это именно та проблема, которую решает snapshot testing в AI-эру. Классические unit-тесты для legacy требуют понять код достаточно хорошо чтобы написать assertions. А snapshot testing говорит: "Не нужно понимать. Просто зафиксируй что есть сейчас, и пусть AI разбирается в диффах".
